# Supabase Guidelines

## Role in the Project

The `supabase/` folder is the **schema source of truth** for the cloud database. It does not run any services — it is a config-and-migrations folder managed by the Supabase CLI and checked into git alongside `frontend/` and `backend/`.

```
The-Nocturnal-Athenaeum/
├── frontend/     ← React/Vite UI
├── backend/      ← FastAPI application
└── supabase/     ← This folder: CLI config + ordered SQL migrations
    ├── config.toml
    ├── docs/
    └── migrations/
        └── 20260613082710_create_profiles.sql
```

---

## The Three Instances

```
frontend  (React/Vite — your machine / deployment)
backend   (FastAPI    — your machine / deployment)
Supabase  (cloud      — Postgres DB + Auth, runs at supabase.com)
```

The `supabase/` folder is **not** a fourth running instance. It is a local project folder that the CLI uses to talk to the cloud instance. Think of it the same way you think of a `.git/` folder — it records state and history, but the actual thing lives elsewhere.

---

## Two CLI Modes — We Use Only One

The Supabase CLI can do two very different jobs. The docs mix them together; this project deliberately picks only one.

| | Local stack (Job 2) | Remote-only (Job 1) ← us |
|---|---|---|
| Where the DB runs | Docker containers on your laptop | The cloud (supabase.com) |
| Needs Docker | **Yes** | **No** |
| Commands used | `supabase start`, `supabase stop`, `db reset`, `db diff` | `login`, `link`, `db push` |
| Good for | Offline dev, teams sharing a throwaway DB | Solo project with one cloud DB |

**Never install Docker or run `supabase start` for this project.** All schema changes go directly to the cloud via `db push`.

## Folder Structure

```
supabase/
├── config.toml        ← CLI config for this project (project_id, auth settings, etc.)
├── docs/
│   └── supabase_guidelines.md   ← this file
└── migrations/        ← ordered, append-only SQL migration files
    └── 20260613082710_create_profiles.sql
```

### `config.toml`

Generated by `supabase init`. The `project_id` field identifies this local folder as belonging to the `The-Nocturnal-Athenaeum` project. Most settings in `config.toml` apply only to the local Docker stack (Job 2) — for Job 1, only `project_id` and auth provider settings are relevant.

### `migrations/`

An ordered, append-only history of every schema change. Each file is applied once to the remote database, in filename order. They are the database's version history — like git commits for the schema.

---

## Migration Conventions

### Naming

```
<14-digit-timestamp>_<description>.sql
```

Example: `20260613082710_create_profiles.sql`

- **Timestamp** — generated automatically by `supabase migration new <name>`. The CLI uses the current datetime so files sort in apply order by name. Never hand-write the timestamp.
- **Description** — snake_case, describes what the migration does at a glance.

### Rules

- **Append-only.** Never edit a migration that has already been applied to the remote. Add a new file for the next change. Editing applied migrations breaks history and will cause `db push` to fail or produce unexpected results.
- **One concern per file.** `create_profiles`, `add_conversations`, `add_messages` — not one giant file for everything.
- **Idempotent where possible.** Use `create table if not exists`, `create policy if not exists`, etc. so that re-runs on a fresh database don't error.

### When to use a migration file vs `models.py`

| Table type | Defined in migration | Defined as ORM model |
|---|---|---|
| Plain app tables (`conversations`, `messages`) | Optional (can use `create_all`) | Yes — SQLAlchemy creates them |
| Anything touching Supabase internals (FK into `auth.users`, RLS policies) | **Required** — SQLAlchemy cannot express these | Mirror only (read/write view) |

The `profiles` table is the second case: the table + FK + RLS live in the migration; `backend/app/modules/auth/models.py` is just the ORM view of it.

## Workflow: Adding a Schema Change

Every schema change follows these four steps:

```
1. Create the file    →  supabase migration new <name>
2. Write the SQL      →  fill in the generated file
3. Review locally     →  read the file, check for typos
4. Push to cloud      →  supabase db push
```

### Step-by-step example: adding a `conversations` table

```bash
# 1. Create the file (CLI sets the timestamp automatically)
supabase migration new create_conversations

# 2. Fill in supabase/migrations/<timestamp>_create_conversations.sql
#    with CREATE TABLE, indexes, RLS, etc.

# 3. Review the file, then push
supabase db push
```

`db push` compares what the remote has already run against all files in `migrations/`, and applies only the new ones.

## One-Time Setup (per machine)

Run these once when setting up the project on a new machine. After this, only `db push` is needed for day-to-day work.

```bash
# 1. Authenticate the CLI to your Supabase account (opens browser)
supabase login

# 2. Link this local folder to the cloud project
supabase link --project-ref vjgkoauuzugfoihmtcbb
#   → prompts for your database password

# 3. Verify the link is working
supabase db push --dry-run
```

The `project-ref` (`vjgkoauuzugfoihmtcbb`) is the ID in your Supabase dashboard URL:
`https://supabase.com/dashboard/project/vjgkoauuzugfoihmtcbb`

## Command Reference

| Command | When to use |
|---|---|
| `supabase login` | First-time setup — authenticates the CLI |
| `supabase link --project-ref <ref>` | First-time setup — connects this folder to a cloud project |
| `supabase migration new <name>` | Every new schema change — creates a timestamped, empty SQL file |
| `supabase db push` | Applies all unapplied migrations to the remote database |
| `supabase db push --dry-run` | Preview what would be applied, without touching the remote |
| `supabase migration list` | Shows which migrations have been applied to the remote |

### Commands to avoid for this project

| Command | Why to avoid |
|---|---|
| `supabase start` | Requires Docker; starts the local stack (Job 2) |
| `supabase stop` | Stops the local stack |
| `supabase db reset` | Wipes and rebuilds the **local** Docker database — does nothing to the cloud |
| `supabase db diff` | Diffs against the local Docker database |
| `supabase db pull` | Pulls remote schema into a new migration file — only needed if you made manual changes in the dashboard outside of migrations |

## What Lives Where

| Concern | Location | Why |
|---|---|---|
| Table + FK into `auth.users` + RLS | `supabase/migrations/*.sql` | SQLAlchemy cannot express cross-schema FKs or RLS |
| ORM read/write view of a table | `backend/app/modules/<module>/models.py` | Python code needs typed models for queries |
| Supabase config (URL, anon key) | `backend/app/core/config.py` + `.env.backend` | All app config lives in `core/`, not scattered |
| Google OAuth provider setup | Supabase dashboard → Authentication → Providers | One-time UI click, not codified |
| JWT verification | `backend/app/modules/auth/service.py` | Delegated to `supabase.auth.get_user(token)` — no secret in config |

---

## What Success Looks Like

A new developer should be able to answer instantly:

- **Where is the schema history?** → `supabase/migrations/`, in filename order.
- **How do I add a new table?** → `supabase migration new <name>`, write SQL, `supabase db push`.
- **How do I see what's applied to the remote?** → `supabase migration list`.
- **Do I need Docker?** → No. This project never uses the local Supabase stack.
